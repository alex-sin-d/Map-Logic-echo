EXERCISE 1

1. What are the base cases of the recursive methods (rfib and rfact)?

For rfib, the base cases are when n == 1 or n == 2, because in both of those cases it just returns 1.
For rfact, the base case is when n == 1, because it returns 1 there.


2. Write down the running time printed out to the console for each of the following inputs:

Factorial(15): 17ms iterative, 0ms recursive
Factorial(25): 1ms iterative, 0ms recursive
Factorial(35): 0ms iterative, 0ms recursive
Factorial(45): 0ms iterative, 0ms recursive

Fibonacci(15): 15ms iterative, 1ms recursive
Fibonacci(25): 0ms iterative, 4ms recursive
Fibonacci(35): 0ms iterative, 41ms recursive
Fibonacci(45): 0ms iterative, 2311ms recursive

3. For which algorithms do the running times grow faster as the size of the input increases?

The recursive Fibonacci algorithm grows way faster than the others. The factorial ones do not grow nearly as badly, and the iterative Fibonacci stays fast.


4. Should variable methodCalls2 be local or an instance variable?

It should not be local. It should be a variable that all recursive calls can access and update, so it should be an instance variable or a class-level variable.


5. What is the value of methodCalls2 (calls to rfib(2)) when n=45?

701408733


6. Why is the recursive algorithm for computing Fibonacci numbers so slow compared to the iterative algorithm? 

It is slow because it keeps recalculating the same Fibonacci values over and over again. The huge value of methodCalls2 shows that the method reaches rfib(2) an insane number of times, which means there is a ton of repeated work. The iterative version just builds up the answer once, so it is much more efficient.

8. How did the execution stack change after you pressed F6 twice?

After the first F6, the print statement in m1 runs, but the stack does not really change.
After the second F6, m1 finishes and its activation record is removed from the stack, so control goes back to m2.
